# Кредитный скоринг  
Нужно предсказать вероятность дефолта клиента по кредиту на основе истории его кредитных продуктов.
Целевая переменная — `flag` (1 — дефолт, 0 — нет).

**Метрика** — ROC‑AUC. Лучший результат на тесте платформы **0.7662**.

Я пошла по пути агрегации признаков на уровне клиента, поскольку каждый клиент имеет несколько записей
Я использовала one‑hot кодирование всех признаков (кроме id и rn) с суммированием по клиенту + добавила среднее, максимум и минимум по всем числовым колонкам. Это позволило сохранить и категориальную информацию, и числовые статистики.
Помимо базовой агрегации, я добавила ранговые признаки: логарифм id, ранг id, ранг количества продуктов и отношение rn_max к числу продуктов. Они должны были помочь модели уловить временные тренды.
Основной упор сделан на XGBoost, CatBoost и LightGBM . Параметры настраивались через Optuna. Затем применяется блендинг моделей

Рассматривась возможность использования FT‑Transformer как альтернативу градиентному бустингу для учёта взаимодействий между признаками, но при попытке обучить FT‑Transformer на наших данных столкнулись с ограничением вычислительных мощностей на устройстве. Обучение трансформера на полном датасете требовало значительно больше памяти, даже при уменьшении размера эмбеддингов и сокращении числа слоёв время обучения составляло несколько часов на один фолд, что не позволяло эффективно экспериментировать.
При обучении нейросети и включения ее в блендинг ожидается повышение ROC‑AUC

В итоге мы отказались от FT‑Transformer в пользу градиентного бустинга
(Выводы в ноутбуке для части данных, а не для всего датасета, поэтому значения метрик отличаются)

Все решение для удобства находится в файле Credit_Scoring(5).ipynb
